Architecture Générale

1. Vision globale détaillée

Le frontend collecte les données utilisateur et les envoie au backend via HTTP. Le backend valide, traite et communique avec la base de données. Chaque couche a une responsabilité unique pour faciliter maintenance et évolutivité.

2. Flux complet

  1. Utilisateur remplit le formulaire
  2. Validation frontend avec Zod pour feedback immédiat
  3. Envoi de la requête HTTP via Axios
  4. Validation backend avec Zod pour sécurité
  5. Insertion sécurisée dans MySQL via requêtes préparées
  6. Réponse JSON renvoyée au frontend

3. Séparation des responsabilités détaillée

  • React → interface utilisateur et gestion état
  • Express → logique métier et routes API
  • Zod → validation cohérente des données
  • MySQL → persistance sécurisée
  • Axios → communication entre front et back

4. Pourquoi doubler la validation

Frontend pour expérience utilisateur et correction rapide, backend pour empêcher toute donnée invalide d’atteindre la base.

5. Architecture logique simplifiée

Client → API → Base de données. Cette structure permet de séparer UI, logique et données.

6. Objectif pédagogique

Comprendre chaque couche isolément, puis voir comment elles s’articulent pour créer une application sécurisée et maintenable.

Express - Création du serveur

1. Rôle technique détaillé

Express crée une API REST pour recevoir et renvoyer des données JSON, centralisant la logique métier et les routes.

2. Activation du mode ESM

{
  "type": "module"
}

Permet d’utiliser les import/export modernes dans Node.js.

3. Serveur minimal

import express from "express";

const app = express();

app.use(express.json());

app.get("/", (req, res) => {
  res.send("API active");
});

app.listen(3000);

4. Explication simplifiée et bonnes pratiques

  • import → charge Express
  • express() → crée le serveur
  • use(express.json()) → middleware pour parser JSON
  • get() → route simple pour test serveur
  • listen() → démarre le serveur sur port défini
  • Bonne pratique : logger chaque requête pour debug

5. Route POST structurée

app.post("/users", (req, res) => {
  console.log(req.body);
  res.json({ message: "Reçu" });
});

6. Vision architecture

Express agit comme couche contrôleur, médiant entre frontend et base de données, permettant une gestion centralisée de la sécurité et des erreurs.

Zod - Validation des données

1. Pourquoi valider

Les utilisateurs peuvent envoyer des données inattendues. Validation indispensable pour éviter corruption de la base et erreurs serveur.

2. Définition d’un schéma

import { z } from "zod";

const schema = z.object({
  name: z.string().min(2, "Nom trop court"),
  email: z.string().email("Email invalide")
});

3. Validation sécurisée

const result = schema.safeParse(req.body);

if(!result.success){
  return res.status(400).json(result.error);
}

4. Différence parse / safeParse

  • parse → lance une exception si invalide
  • safeParse → renvoie objet avec success=true/false, pas de crash serveur

5. Intérêt stratégique

Prévenir l’insertion de données incorrectes et standardiser le format côté backend et frontend.

MySQL - Gestion des données

1. Rôle

Stocker de manière fiable et sécurisée les données validées.

2. Connexion via pool

import mysql from "mysql2/promise";

const pool = mysql.createPool({
  host:"localhost",
  user:"root",
  password:"password",
  database:"test"
});

3. Requête sécurisée

await pool.query(
 "INSERT INTO users (name,email) VALUES (?,?)",
 [name,email]
);

4. Pourquoi les ?

Évite l’injection SQL et protège les données sensibles.

5. Cycle BDD

  1. Connexion
  2. Exécution requête préparée
  3. Retour des résultats

6. Bonnes pratiques

  • Ne jamais exposer les informations de connexion côté frontend
  • Logger erreurs de manière contrôlée
  • Valider les données avant insertion

React Hook Form + Zod

1. Gestion des formulaires

Permet de gérer les états et la validation de formulaires complexes sans re-render excessif.

2. Schéma Zod côté frontend

const schema = z.object({
  name: z.string().min(2, "Nom trop court"),
  email: z.string().email("Email invalide")
});

3. Formulaire complet

const { register, handleSubmit, formState:{errors} } =
useForm({ resolver: zodResolver(schema) });

const onSubmit = data => console.log(data);

4. Comprendre chaque élément

  • useForm → moteur de formulaire
  • register → relie input au formulaire
  • handleSubmit → déclenche validation et soumission
  • errors → messages d’erreur

5. Objectif UX

Valider côté frontend pour retour rapide, sans se substituer à la validation backend.

Intégration complète avec Axios

1. Création d'une instance Axios

L'instance Axios permet de centraliser la configuration pour toutes les requêtes HTTP. Cela facilite les modifications et assure cohérence dans toutes les requêtes.

import axios from "axios";

// Création d'une instance avec baseURL
const api = axios.create({
 baseURL: "http://localhost:3000", // URL du backend
 timeout: 5000, // délai maximum avant timeout
 headers: {
   "Content-Type": "application/json" // définit le type de données
 }
});
  • baseURL : point d'entrée de votre API
  • timeout : évite que la requête bloque trop longtemps
  • headers : standardisation des entêtes HTTP

2. Envoi POST pas à pas

Exemple d'envoi de données vers le backend avec gestion des erreurs.

try {
  const response = await api.post("/users", {
    name: "Alice",
    email: "alice@example.com"
  });

  console.log("Réponse serveur :", response.data);

} catch (error) {
  if (error.response) {
    // Le serveur a répondu avec un code d'erreur
    console.error("Erreur serveur :", error.response.status, error.response.data);
  } else if (error.request) {
    // La requête a été envoyée mais pas de réponse
    console.error("Pas de réponse du serveur", error.request);
  } else {
    // Erreur lors de la configuration de la requête
    console.error("Erreur Axios :", error.message);
  }
}
  • try/catch : gestion globale des erreurs côté frontend
  • error.response : erreur renvoyée par le serveur
  • error.request : aucune réponse reçue
  • Permet d’afficher un message utilisateur approprié

3. Cycle complet de l'intégration

  1. L'utilisateur soumet le formulaire
  2. React récupère les valeurs et les valide via Zod
  3. Axios envoie une requête POST au backend
  4. Express reçoit la requête et valide à nouveau les données
  5. MySQL insère les données avec requêtes préparées
  6. Le serveur renvoie un JSON confirmant l’insertion
  7. Frontend affiche un message de succès ou d'erreur

4. Rôle des modules

  • React : interface utilisateur, collecte et affichage des données
  • React Hook Form : gestion efficace des états et validation locale
  • Zod : validation stricte et cohérente côté frontend et backend
  • Axios : communication HTTP entre front et back
  • Express : serveur API REST, validation et traitement métier
  • MySQL : stockage sécurisé des données

5. Gestion des erreurs et bonnes pratiques

  • Afficher des messages clairs à l'utilisateur
  • Logger côté serveur sans exposer d'informations sensibles
  • Timeout et retry en cas de connexion lente
  • Ne jamais faire confiance aux données frontend, valider à chaque étape

Sécurité simplifiée

1. Pourquoi sécuriser

La sécurité protège votre application et vos utilisateurs. Même si le frontend contrôle les données, le backend doit toujours vérifier pour éviter les erreurs et les attaques.

2. Validation des données

Vérifiez toujours les données envoyées par l’utilisateur :

  • Frontend : retour rapide pour l’utilisateur (ex : champs obligatoires, format email)
  • Backend : validation obligatoire avant d’enregistrer dans la base

3. Protéger les données sensibles

Quelques règles simples :

  • Ne jamais mettre les mots de passe ou clés secrètes dans le code visible
  • Utiliser un mot de passe sécurisé pour la base de données
  • Limiter les informations envoyées à l’utilisateur en cas d’erreur

4. Communiquer de manière sécurisée

  • Utiliser HTTPS pour chiffrer les échanges entre frontend et backend
  • Limiter la taille des formulaires pour éviter les surcharges
  • Ajouter un contrôle simple de nombre de requêtes pour éviter les abus

5. Bonnes pratiques simples

  • Valider toutes les données côté backend
  • Utiliser des requêtes préparées pour la base (empêche l’injection SQL)
  • Afficher uniquement des messages génériques en cas d’erreur côté utilisateur
  • Logger les erreurs côté serveur pour les développeurs

Exercices pratiques détaillés

Exercice 1 – Module produit

Énoncé : Créez un module complet permettant d’ajouter des produits dans la base. Le formulaire doit contenir :

  • Nom du produit (minimum 3 caractères)
  • Prix (nombre positif)
  • Description (minimum 10 caractères)

Le backend doit valider avec Zod et insérer les données dans MySQL via requêtes préparées. Le frontend doit gérer la validation et l’affichage des erreurs.

Exercice 2 – API sécurisée commandes

Énoncé : Créez une route POST /orders permettant d’enregistrer des commandes. Chaque commande doit contenir :

  • ID utilisateur
  • Liste des produits commandés
  • Quantité par produit
  • Date de commande

Le backend doit :

  • Valider la structure des données avec Zod
  • Utiliser des requêtes préparées pour MySQL
  • Renvoi un code 400 si les données sont invalides
  • Tester différents scénarios d’erreur et succès

Exercice 3 – Formulaire avancé React

Énoncé : Créez un formulaire de produit avancé avec React Hook Form et Zod :

  • Ajoutez un champ productType : gratuit ou payant
  • Si productType = payant, le prix doit être > 0
  • Validez côté frontend et backend
  • Affichez des messages d’erreur clairs côté frontend
  • Testez insertion correcte dans MySQL
